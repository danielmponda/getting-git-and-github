Introduction

One can certainly write some reasonably large and complex programs by themselves - however, depends on the scale and complexity of the project, software development can take weeks, months or even years and can’t simple be done by one person. To complete and distribute a given project in a reasonable amount of time, you may need more than one person working on a given project. Today, a team of developers is responsible for most software builds. For instance, both Google and Facebook have hundreds of software Engineers working and running their respective sites simultaneously. To commodity this many software Engineers working concurrently on a single project can be problematic.

Just to mention some of the challenges.

- With hundreds of developers working on a single project simultaneously, how do we keep track on who made a changed, what have been changed and when did the change occur?

- How to Integrate and synchronize the work that is done simultaneously by different members of the team. Especially when conflicting edits are made by two people to the same line of a file.

- How do we keep history of changes and historical versions of a project in case something goes totally wrong?

The solution to this is to develop a system which could provide a logical method that preserves the individual contributions of team members without overwriting another developer's work, and then merges and reconciles changes and updates. The systems should be able to produce a historical record which can be used to track down problem origins or archive versions for later use, this is what Version Control Systems attempts to do.

Version Control Systems:

Version control systems are a category of software tools / utility that tracks and manages changes to a filesystem, it provides collaborative utilities to share and integrate these filesystem changes by those contributing to a given project. Version control systems will track the addition, deletion, and modification actions applied to files and directories. When a filesystem is traced this is refereed to as a repository.

Types of Version Control Systems:

Centralized Version Control Systems: Centralized version control systems contain just one repository and each user gets their own working copy. You need to commit to reflecting your changes in the repository. It is possible for others to see your changes by updating

Distributed Version Control Systems: Distributed version control systems contain multiple repositories; it consists of local and remotely repositories. Each user has access to their own local copy of the entire repository and to central remotely repository. Unlike the centralized version control, this means that the code is not just stored in a central server, but the full copy of the code is present in all the developers’ computers.

Git and GitHub

Git being a distributed version control system, has a remote repository which is stored in a server and a local repository which is stored in the computer of each developer. This means that the code is not just stored in a central server, but the full copy of the code is present in all the developers’ computers. in order to have a central server repository you need firm that from hosting server for your central server repository, luckily there are companies out there who are willing to host your central repositories. Such companies include, GitHub and Bitbucket.

GitHub, is simply an American multinational corporation that provides hosting for software development and version control using Git for free. It offers the distributed version control and source code management (SCM) functionality of Git, plus its own features.

In order to use Git and GitHub in your project, you first need to download Git to track all your filesystems using your local repository and set up a remote repository on the GitHub website.

Installation

Have you successfully installed Git?

Use the following command in the command prompt to Verify

git –version

Initialize

Are you currently in the project folder?

Use the following command in the command prompt to change directory.

cd myproject

to initialize Git in a new or an existing project on your local machine

Use the following command in the command prompt.

git init

Tracking Changes

Have you made, deleted, or modified any files or directories? i.e index.html

Use the following command for Git to add the change to the Staging Area.

git add index.html
Staging Area holds a snapshot of the content of the working tree, and it is this snapshot that is taken as the contents of the next commit.

Satisfied with your changes you added to the Staging Area ?

Use the following command to record and comment on the chances to the local repository

git commit -m "Initial Commit Message"

Not satisfied with the recent commit?

Use the following command to revert your code based on commit ID

git reset --hard commit ID

Cant remember the commit ID ?

Use the following command to list all commit info

git log

Publishing

Ready to publish your changes to a remote repository ?

Use the following command to connect to a remote repository.

git remote add

Have you successfully connected to a remote repository ?

Use the following command to push all the code from the local repository into the remote repository.

git push -u origin master

Branch

Do you want to isolate from original code and work on new or particular feature/s ?

Use the following command to create and branch off the master .

git branch branch-name

Use the following command to switch to the new created branch .

git checkout newBranch

Satisfied with the new advancement features ?

Use the switch command to head back the master branch.

git checkout master

Use the following command to merge the newBrach to the master branch.

git merge newBranch
